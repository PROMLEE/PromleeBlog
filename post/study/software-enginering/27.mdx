## GRASP란 (What is GRASP?)
**G**eneral **R**esponsibility **A**ssignment **S**oftware **P**atterns
- 일반적인 책임 할당 소프트웨어 패턴

OOAD로 소프트웨어를 구성할 때의 일반적인 작업
- 요구 사항 식별
- 도메인 모델 생성
- *디자인 모델 생성*
	- *소프트웨어 클래스에 속성과 메서드 추가*
	- *객체 간의 상호 작용 정의*
<br/>

GRASP는 **협력 객체**에 **책임을 할당**하는 방법을 제시

## 책임 (Responsibility)
> 클래스의 계약 또는 의무

**아는 책임**(knowing responsibility)
- 비공개 캡슐화 데이터에 대해 알고 있는 것
- 관련 객체에 대해 알고 있는 것
- 파생되거나 계산될 수 있는 것에 대해 알고 있는 것
- 예) "판매(Sale)"는 총액을 아는 책임이 있음

**하는 책임**(doing responsibility)
- 스스로 어떤 일을 수행하는 것
- 다른 객체에서 동작을 시작하는 것
- 다른 객체에서 활동을 제어하고 조정하는 것
- 예) "판매(Sale)"는 판매 항목을 생성하는 책임이 있음

## 책임 할당 일반 원칙 (General Principles of Assigning Responsibilities)
1. **Information Expert**
2. **Creator**
3. **Controller**
4. **Low Coupling**
5. **High Cohesion**
6. **Polymorphism**
7. **Pure Fabrication**
8. **Indirection**
9. **Protected Variations**

## 모듈성 (Modularity)
설계 목표
- 시스템을 모듈로 분할하고 구성 요소 간에 책임 할당

모듈성은 프로그래머가 한 번에 처리해야 할 전체 복잡성을 줄여준다
- 기능을 유사한 기능끼리 묶어 모듈에 할당한다 (*관심사의 분리*)
- 모듈 간의 작은, 간단하고 잘 정의된 인터페이스가 있다 (*정보 은닉*)

모듈성 측정
- 결합도: 모듈 간 *상호 의존성*의 측정
- 응집도: 모듈 내 요소들이 얼마나 강하게 *관련되어 있는지*의 측정

## 결합의 개념 (Concept of Coupling)
**결합**
- 클래스(또는 모듈) 간 상호 의존성의 측정
- 높은(또는 강한) 결합을 가진 클래스는 많은 다른 클래스에 의존한다
- 독립된 요소가 변경될 때, 그것이 의존하는 요소에 영향을 미친다
	<Img src="/posts/240617-232229.png" width={400}/>

*낮은 결합 추구할 것*
높은 결합의 단점
- 의존 클래스의 변경으로 인한 영향
- 독립적으로 이해하기 어려움
- 재사용성이 낮음

## 객체 지향 언어에서의 결합 형태 (Types of Coupling in Object-Oriented Languages)
Type X가 Type Y에 의존하는 경우
- TypeX가 *TypeY 인스턴스* 또는 TypeY 자체를 참조하는 속성을 가지고 있다
- TypeX가 어떤 방법으로든 TypeY 인스턴스 또는 TypeY 자체를 *참조하는 메서드*를 가지고 있다 (예: TypeY 타입의 매개변수 또는 지역 변수, 반환된 객체가 TypeY 인스턴스인 경우 등)
- TypeX가 TypeY의 직접 또는 간접 *서브클래스*이다
- TypeX가 TypeY 인터페이스를 *구현*한다
- TypeX가 TypeY 인터페이스를 *사용*한다 (예: TypeY 인터페이스를 매개변수로 받는 메서드)
	<Img src="/posts/240617-232608.png" width={500}/>
	- **직접 참조**: solid line으로 표시하며, TypeX가 TypeY를 직접 참조하는 경우이다.
	- **메서드 참조**: dotted line으로 표시하며, TypeX가 메서드에서 TypeY를 참조하는 경우이다.
	- **상속**: solid line with a hollow arrowhead로 표시하며, TypeX가 TypeY를 상속받는 경우이다.
	- **인터페이스 구현**: dashed line with a hollow arrowhead로 표시하며, TypeX가 TypeY 인터페이스를 구현하는 경우이다.

## 응집의 개념 (Concept of Cohesion)	
> 모듈 내 요소들이 얼마나 강하게 관련되어 있는지의 측정

관련된 책임들이 하나의 관리 가능한 단위로 배치된다
단일 책임 원칙(SRP)와 관련이 있다
<br/>

**높은 응집도를 추구할 것**
높은 응집도의 장점
- 쉽게 이해하고 유지보수할 수 있다
- 재사용성이 높다
- 낮은 결합도를 가진다
## GRASP 원칙 패턴 (GRASP Patterns)
### 창조자 (Creator)
> 클래스 간의 책임 할당을 정의하는 GRASP의 원칙 중 하나로, 특정 클래스의 *인스턴스를 생성할 책임*을 어떤 클래스에게 할당할지 결정하는 가이드라인을 제공한다.

### 정보 전문가 (Information Expert)
> 클래스 간의 책임 할당을 정의하는 GRASP의 원칙 중 하나로, 특정 작업을 수행하는 데 *필요한 정보*를 가지고 있는 클래스에게 책임을 할당하는 가이드라인을 제공한다.

### 컨트롤러 (Controller)
> 클래스 간의 책임 할당을 정의하는 GRASP의 원칙 중 하나로, *사용자 인터페이스 이벤트를 처리하고 시스템 작업을 시작하는 책임*을 어떤 클래스에게 할당할지 결정하는 가이드라인을 제공한다.

### 낮은 결합도 (Low Coupling)
> 클래스 간의 책임 할당을 정의하는 GRASP의 원칙 중 하나로, *클래스 간의 상호 의존성을 최소화*하는 가이드라인을 제공한다.

### 높은 응집도 (High Cohesion)
> 클래스 간의 책임 할당을 정의하는 GRASP의 원칙 중 하나로, *클래스 내부의 요소들이 얼마나 강하게 관련되어 있는지*를 측정하는 가이드라인을 제공한다.

### 순수 가공 (Pure Fabrication)
> 클래스 간의 책임 할당을 정의하는 GRASP의 원칙 중 하나로, *책임을 할당할 수 있는 적절한 클래스가 없을 때* 책임을 할당할 클래스를 생성하는 가이드라인을 제공한다.

### 다형성 (Polymorphism)
> 클래스 간의 책임 할당을 정의하는 GRASP의 원칙 중 하나로, *객체의 타입에 따라 다르게 동작*하는 책임을 어떤 클래스에게 할당할지 결정하는 가이드라인을 제공한다.

### 간접 (Indirection)
> 클래스 간의 책임 할당을 정의하는 GRASP의 원칙 중 하나로, *클래스 간의 직접적인 상호 작용을 최소화*하는 가이드라인을 제공한다.

### 변화 보호 (Protected Variations)
> 클래스 간의 책임 할당을 정의하는 GRASP의 원칙 중 하나로, *변화에 대한 보호*를 제공하는 가이드라인을 제공한다.
